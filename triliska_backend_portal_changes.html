Triliska backend portal additions and changes for programme staff

These are the modules recommended to add to the web backend portal so programme staff can prepare an MSME's full setup from a desktop — before handing over the POS device to the trader. All of these are staff-assist features; the business owner takes over once onboarding is complete.

📦
Product master
  • Create / edit products per MSME
  • Set name, SKU, barcode, price, cost, tax flag
  • Assign product category and unit of measure
  • Expiry date field (pharmaceutical sector)
  • CSV bulk import with validation error report
  • Download product template CSV
  • Deactivate / archive products
Who uses it: Programme Staff, Admin
🏪
Stores & warehouses
  • Create named store / warehouse locations per MSME
  • Set default store for POS
  • View stock levels per store
  • Record inter-store stock transfers
  • Mark stores as active / inactive
Who uses it: Programme Staff, Admin
📊
Opening stock setup
  • CSV bulk opening stock import per MSME
  • Manual opening stock entry per product
  • Assign opening stock to specific store
  • Download stock template CSV
  • View stock import history and errors
Who uses it: Programme Staff, Admin
✅
Onboarding checklist
  • Per-MSME progress tracker (% complete)
  • Steps: Registered → Profile complete → Products loaded → Stock set → Device linked → First sale
  • Flag blocked items needing follow-up
  • Staff can mark steps complete with notes
Who uses it: Programme Staff, Supervisors, Admin

  • View last sync timestamp per device

  • Unlink / reassign device

Why add product master to the backend portal — not just the POS?
Programme staff working on a desktop have a full keyboard, large screen, and can handle a multi-row CSV with hundreds of products efficiently. Doing the same task on a 5.99" Sunmi screen is slow, error-prone, and unsuitable for bulk setup. The web portal also lets staff pre-load an MSME's full product catalogue before visiting the market — the trader opens their POS and everything is already there. This removes a significant onboarding bottleneck and means the first visit is focused on training, not data entry.
Remove Items to remove from the current backend
✕
Payments nav item — hide from all non-owner views
Only Business Owners can initiate tax payments. Remove the Payments section (including the "Pay Now" button) from Admin, Programme Supervisor, Programme Staff, and GRA views. Admin may see payment history for oversight only — no initiation capability.
AdminSupervisorProgramme StaffGRA
✕
Submissions section — disable until middleware endpoint is defined
The current Submissions screen has unclear requirements, mixes Odoo Import and CSV source types, and has no defined endpoint from the Aliska middleware. Disable the nav item and surface a placeholder until the submission flow is formally specified. Data should flow directly from POS to the backend database — no manual re-submission to GRA.
✕
SGD currency — replace with GHS throughout
All monetary values display as SGD (Singapore Dollar). Must be replaced with GHS (Ghana Cedis, GH₵) across the dashboard, payments screen, submissions, and all reports. This is a correctness issue — the system is deployed for Ghana.
✕
"admin" label in top-right — rename to Programme Admin
The top-right user pill shows the generic string "admin". It should display the logged-in user's full name with their role badge (e.g. "Robert Nettey · Programme Admin"). Avoids confusion with business-level admin concepts.
Add Items to add — programme operations
+
Role-scoped dashboard — different views per role
The current dashboard is a single shared view. It must be role-aware: Admin sees programme-wide KPIs. Supervisor sees their staff's companies and team metrics. Business Owner sees business performance (revenue, stock alerts, tax due). GRA Staff sees compliance and submission summary only.
+
Onboarding progress tracker on dashboard (Admin + Supervisor)
KPI cards showing the pipeline: total MSMEs registered → products loaded → inventory set → POS device linked → first sale recorded. Shows how many MSMEs are at each stage. Critical for tracking pilot readiness at Makola Market.
+
Programme Staff section — assignments, activity, performance
A dedicated section for Admins and Supervisors to: view all programme staff, see company assignments per staff member, track visit logs, and review staff performance KPIs (MSMEs onboarded, products loaded, escalations pending).
+
Zone / district filter on Companies list and dashboard
Companies list must be filterable by district and zone. Dashboard KPI cards must auto-scope to the logged-in user's zone. Programme Supervisors should only see MSMEs under their assigned staff — this must be enforced at the API level, not just the UI.
+
Tax compliance status column on Companies list
A status column per MSME showing: Not started / Products loaded / Sales active / Submission pending / Up to date. Gives instant operational status without drilling into each record. Colour-coded: grey → amber → green progression.
+
GRA Staff dashboard — compliance reporting view
Dedicated read-only view for GRA Officers: total registered MSMEs, MSMEs with pending submissions, total tax collected to date, download reports by period / district / sector. No business operations data visible to GRA.

Clarify Items needing a decision before building
?
Submission flow — automated or manually triggered?
Data should go directly from POS to the Triliska database. But who triggers the GRA submission — does the system generate it automatically on a monthly schedule, or does the Business Owner manually confirm each submission from the portal? The current demo shows a manual button. Confirm the intended flow with Aliska before building the Submissions module.
?
POS sync target — custom backend or Odoo?
The backend currently shows "Odoo Import" as a submission source type. The architecture decision (PRD v4.2 Odoo path vs TRD v1.2 custom FastAPI path) is still open. Product master, inventory, and stock modules must be built against the agreed data store. This decision must be made before Sprint 1B begins.
